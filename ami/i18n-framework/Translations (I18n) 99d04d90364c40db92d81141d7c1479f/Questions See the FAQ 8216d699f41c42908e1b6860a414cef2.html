How does a user's locale/language get set?

Localization is enabled through a system-wide feature flag and thus not everyone can use it. Now, if the feature flag is enabled, we'll will try to set the app's language to the user's preference, which they can configure in account settings. If not available, we'll set it to the user's primary membership supplier market locale. If the user doesn't have a supplier market, then we'll set it to the default, English.

Nonetheless, there is a small exception. If the feature flag is enabled at 100%, then we'll set the language to the OS/browser's locale just before falling back to English.

I need to change the name for a key on Lokalise (typo?), but it doesn't allow me to do so. What do I do?

We've blocked renaming keys, as this might break production and other environments due to non-existing key references. However, if you do need to change the key name (because of a typo, refactor, follow-conventions, etc.), create a new one. And label the wrong one with "delete-me-later". Admins (or maybe automation) will eventually delete those ones, after ensuring there's no references to such keys in the code.

I changed the value of a translation. Why are my changes not being displayed in Production?

  • It takes up to 1 hour for changes to propagate to S3.
  • View might be cached. Not much to do in PRD, just wait. On rev/stg, a Rails.cache.clear will help you.
  • Dynos might not have been restarted, and pre-exiting translations are memoized in-memory. Restart dynos if truly needed or wait 1 day, as dynos get restarted daily.
  • Frontend caching (React) lasts for 1 day. Currently, there's no way to expire it earlier.

How do I add a new language?

  • Add the language in Lokalise.
  • Add a new key in Lokalise with format languages.my_new_locale (ex. languages.fr_CA) and translate this value to all languages, including this new language you are adding (i.e Canadian French, Frances Canadiense, Canadien Français, etc...). This is used for the "Select language" dropdown.
  • Translate Ordinals (1st, 2nd, 3rd, etc...)

    Unfortunately, ordinals must be translated manually in Ruby locale files (yes, not YAML) by a developer, as their logic can be quite complicated. The don't live in Lokalise. See config/locales/fr_CA.rb in Maven for an example. I'm not sure if there are languages without ordinals, but if you find such case, just add lambdas in the respective locale rb file that return the number, such that english are not used as fallback.

  • Add the language to I18n.available_locales in config/initializers/i18n_active_record.rb. If it's a base language (not a supplier-specific language), add it to Translation::BASE_LOCALES in the same file.
  • Configure fallbacks in config/application.rb if needed (a must for non-english-based supplier-specific languages). Search for config.i18n.fallbacks to see how they are configured, and append the new chain to the hash.

How does genderization work?

Genderization will have happen automagically when the Maven::I18n.t or mt methods are used in Ruby-land, or when the useTranslation or withTranslation from maven/i18n are used in JS-land. Both of this rely on the gender or gender_guess of the Current.user, which is a global set in all controller actions of the seller app. This is good enough for most cases, as the subject interface string refer to is the app's user. However, you should also pay attention to the subject of the sentence as it might not be the Current.user. For example, when there is a contact referenced: Aleksandra just purchased a pina colada. If translating this to Russian, the sentence need to be genderized according the gender of the name of the contact (in this case Alex. Hence, you will need to manually specify it on the Maven::I18n.t helper like this: 

user = User.find_by(first_name: 'Aleksandra')
Maven::I18n.t('contact.purchase', contact: user.first_name, gender: user.gender || user.gender_guess)
# => Александра только что купила пина коладу
user = User.find_by(first_name: 'Alex')
Maven::I18n.t('contact.purchase', contact: user.first_name, gender: user.gender || user.gender_guess)
# => Алекс только что купил пина коладу

There's currently no support for manually specifying gender on the JS-side, but it should be implemented when and if the need arises.

Both helpers pick the proper genderized translation by selecting .male or .female suffixed keys when the gender is provided (implicitly or explicitely)

I've seen we have some locale/verbiage files in the codebase. Do we still need to use them?

Locale files should no longer be used. They will be removed eventually. These include all those living under config/locale or under the verbiage folder in todos.

I’m not seeing updated translations on iOS/Android?

Some translations strings are handled natively (i.e. they are not translated by our Rails app), and unlike our Rails app, updates are not reflected automagically. To update this native strings, you need to build “bundles” in Lokalise, for each platform. These are the steps to take:

  1. Go to https://app.lokalise.com/download/5138769260f09acc1f2200.26839545/
  1. Set Format to Lokalise Android SDK
  1. Set Include tags to android
  1. Select all supported languages in the right hand side (if not already)
  1. Click on Build Only.
  1. Repeat steps 1-5, but this time for iOS, so replace Format with Lokalise iOS SDK and Include tags with ios and build!
  1. Then go https://app.lokalise.com/settings/5138769260f09acc1f2200.26839545/#ios
  1. Name the latest build (you will see as created a few moments ago) and mark it as the production and prerelease one.
  1. Save changes.
  1. Repeat steps 7-9, but for android build on https://app.lokalise.com/settings/5138769260f09acc1f2200.26839545/#android
