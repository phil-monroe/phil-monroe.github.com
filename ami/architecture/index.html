AMI Platform Architecture

High level overview

The API Platform leverages a micro-service architecture with multiple Platform/Infrastructure as a Service providers to provide our partners with a fast, reliable, and secure product.

AMI Applications

1. Maven

Maven is the primary application that powers the AMI platform. It coordinates all of the interactions between sellers, buyers, and partner storefronts.

Responsibilities

  • Host seller accounts for seller mobile app and desktop portal
  • Host seller social storefronts for buyers to interact with seller content and links.
  • Host partner dashboard for reporting and configuration
  • Host partner onboarding websites
  • Host AMI admin interface

2. Autobot

The Autobot service is the integration hub that connects the AMI platform to various external partners and services. Autobot provides consistent internal APIs for all custom partner/service APIs.

Responsibilities

  • Persist raw, external data in a structured, searchable format.
  • ETL data from various partner and service datasources (APIs, File Uploads, Web Scrapes, etc)
  • Provide Maven application with structured data for internal data representations.
  • Allow custom, configurable code for platforms and partner specific interfaces.

3. Storefront

The Storefront service provides dedicated APIs and configurable Javascript SDKs for partner storefronts. It is built for speed, cache-ability, and reliability of data for any data required for use in partner storefront frontends.

Responsibilities

  • Serve configurable Javascript SDKs for direct inclusion in partner storefront frontends (if required)
  • Provide highly cached and reliable data for partner storefront frontends

4. Images

The Images service stores, transforms, and serves every piece of media imported into the AMI platform. This allows content loaded into the AMI platform to remain visible and unbroken, even if the original asset URLs change or break over time.

Responsibilities

  • Download and persist all referenced images from a URL into an AWS S3 bucket
  • Transform images based on image type to maximize performance and reliability
  • Serve images optimized for cache-ability with our CDN.

AMI Client Applications

All native mobile applications leverage the server side Rails functionality to deliver the application content and features. The native clients only provide hooks into platform specific features that require platform specific code and leverage https://github.com/turbolinks/turbolinks and its corresponding native libraries to display content in app.

iOS App

Uses the https://github.com/turbolinks/turbolinks-ios project for displaying server side Rails content in the native application.

Android App

Uses the https://github.com/turbolinks/turbolinks-android project for displaying server side Rails content in the native application.

Web Browser

Generally a traditional Ruby on Rails experience with React components used throughout.

Common Patterns for Reliability used at AMI

  • All web servers are autoscaled via Rails Autoscale based on traffic
  • All worker servers are autoscaled via Rails Autoscale based on queue latencies
    • Depending on the project, multiple worker types may be configured for independent autoscaling based on core responsibilities (document indexing, user to dos, metrics, integrations, etc.).
  • PostgreSQL databases run with a Primary-Replica pattern for horizontal scalability (as needed)
  • All PostgreSQL databases support point-in-time recoveries
  • All Redis caches and databases run with a Primary-Replica pattern for high availability
  • Edge networks provide CDN caching, Web Application Firewalls (WAF), security monitoring, and DDoS protection.
  • All applications are monitored using Bugsnag for error reporting and Logz.io for logs
  • All services provide realtime alerting to our On-Run engineers through PagerDuty
  • All PostgreSQL databases are monitored using pganalyze
